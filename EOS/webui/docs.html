Quick Start

Prerequisites

Python 3.11 must be installed before running anything else.

1
Install Python 3.11
Download from python.org/downloads. During installation, check "Add Python to PATH" — this is required.
2
Confirm Python is available
Open a command prompt and run:
python --version
If you see a version number, you're good. If you get "not recognized", re-install Python and check the PATH option.
Quick Start

Run Setup & Verify

Setup downloads all models and binaries. Verify confirms everything landed correctly.

1
Run Setup-Full.ps1
Right-click Setup-Full.ps1 → Run with PowerShell.
If Windows asks about execution policy, click Open or type Y when prompted.
This downloads approximately 13 GB of models and binaries. It only needs to run once.
2
Verify the install
Open a command prompt in the EOS folder and run:
python verify.py
This checks Python packages, binaries, model files, port availability, and config.json validity. Fix any errors it reports before continuing.
Setup script options
📦
Setup-Full.ps1
Downloads everything — all models, binaries, and Python packages. Use this if you want EOS to just work.
🪶
Setup-Lite.ps1
Downloads support assets and packages only. Use this if you want to supply your own large model files.
Quick Start

Launch EOS

EOS startup is always two steps: start the backend servers, then start EOS itself.

1
Start the backend servers
For the recommended stack, run start-standard.bat.
This opens separate windows for the main model (8080), tool helper (8082), and thinking helper (8083).
Wait for each window to print a ready message before the next step.
2
Start EOS
Run start-eos.bat. EOS discovers the running services and prints a capability summary:
Main model:       active
Tool helper:      active
Thinking helper:  active
Creativity:       unavailable (fallback to main)
Vision:           unavailable
STT:              active
TTS:              active

chat: available  |  tools: available  |  voice: available
When it prints Starting WebUI at http://127.0.0.1:7860/, EOS is ready.
3
Open the interface
Navigate to http://127.0.0.1:7860/ in your browser.
The admin panel is at http://127.0.0.1:7860/admin.
💡
Status check anytime
Run status-eos.bat or python eos.py --status to see which services are active, degraded, or unavailable without restarting anything.
🛑
Stopping EOS
Press Ctrl+C in the start-eos.bat window, or close it. Close any backend launcher windows to stop those model servers.

Capabilities Panel

The top of Control & Permissions has live toggles for every major capability. Changes take effect immediately — no restart. They reset to config.json values on next boot. To make a change permanent, edit config.json.

🧠
Autonomy
Four dimensions: Perception, Cognition, Action, Initiative. Each independently toggleable. See the Autonomy section for what each one controls.
🖥
Computer Use
Enable/disable the subsystem and set the operating mode: Off, Command Only, or Supervised Session.
📁
Workspace
Toggle Allow Delete and Allow Exec separately. Read and write are always on.
✨
Creativity
Enable/disable, set injection frequency (Off/Low/Medium/High), and toggle Autonomous Idle separately.
📧
Google
Gmail Send toggle (off by default, marked ⚠). Sending is the only runtime toggle — other Google settings require config.json + restart.
User Guide

Autonomy Dimensions

EOS has four independently controllable dimensions that gate what it can do. Toggle each live from the Capabilities panel.

👁 Perception
Observing the environment — screen capture, sensor data. Turn off to prevent EOS from seeing your screen even when computer use is on.
🧠 Cognition
Background thinking, reflection loops, and self-evaluation between turns. Turn off for a purely reactive, no-background-activity assistant.
🔧 Action
Using tools — file operations, web requests, Discord, Google, computer use. Turn off to make EOS text-only and unable to interact with external systems.
🎯 Initiative
Acting without being asked — proactive messages, autonomous idle behavior, unsolicited task execution. Turn off for a fully passive assistant that only responds when spoken to.
💡
Recommended settings
For full testing: all four on. For a safe passive assistant: turn off Action and Initiative. For read-only access (can observe but not act): Perception on, Action off, Initiative off.

Discord Bot

EOS joins your Discord server as a bot. By default it only responds when directly mentioned (@BotName). It supports full conversations, and slash commands including !initiative and !status.

📁
Where to put the credential
Create a plain text file at: AI personal files\Discord.txt

The file contains only your bot token on the first line — nothing else. That folder already exists in the EOS directory. If you don't see it, create it.
How to get a Discord bot token — step by step
1
Create a bot application
Go to discord.com/developers/applications → New Application. Give it a name — this becomes the bot's display name in Discord.
2
Get the bot token
In the left sidebar click Bot. Click Reset Token and copy it immediately. You only see this once. Paste it into AI personal files\Discord.txt right now.
3
Enable Message Content Intent
Still on the Bot tab, scroll to Privileged Gateway Intents. Enable Message Content Intent. Without this, EOS cannot read what users write.
4
Invite the bot to your server
Go to OAuth2 → URL Generator. Under Scopes check bot. Under Bot Permissions check Read Messages/View Channels, Send Messages, Read Message History. Copy the generated URL, open it in your browser, select your server, and click Authorize.
5
Confirm your config
In config.json confirm this block is present and enabled:
"discord": {
    "enabled": true,
    "credential_file": "AI personal files/Discord.txt",
    "respond_only_to_mentions": true,
    "ignore_bots": true
}
Set "respond_only_to_mentions": false to have EOS respond to all messages in any channel it can read. Use with caution in busy servers.
6
Restart EOS
The Discord bot starts automatically at boot once the credential file is in place and enabled is true. No additional steps needed.
🤖
Need a walkthrough? Copy this prompt and paste it into any AI assistant:
"Walk me through creating a Discord bot application in the Discord Developer Portal on Windows — I need to get the bot token, enable the Message Content Intent under Privileged Gateway Intents, and generate an invite URL with bot scope and read/send message permissions. Explain each step clearly."
⚠️
Treat the token like a password
Your bot token grants full control over the bot and access to every server it has joined. If it leaks, reset it immediately in the Discord Developer Portal under the Bot tab.

Google Workspace

EOS can read your Google Calendar, search Gmail, and browse Google Drive files. When enabled, it can also send email. Ask EOS things like "what's on my calendar tomorrow" or "find the email about the project deadline."

📁
Where to put the credential
Place your downloaded OAuth JSON file in: AI personal files\

The filename must match the pattern client_secret_*.json. The file Google gives you is already named correctly — do not rename it, just move it into that folder.
How to get a Google OAuth credential — step by step
1
Create a Google Cloud project
Go to console.cloud.google.com. Click the project selector at the top → New Project. Give it any name (e.g. "EOS Integration") and click Create.
2
Enable the APIs you want
Go to APIs & Services → Library and enable these (search by name, click each, click Enable):
Google Calendar API
Calendar read access
Gmail API
Email read and send access
Google Drive API
Drive file access
3
Configure the OAuth consent screen
Go to APIs & Services → OAuth consent screen:
1. Choose External user type → Create
2. Fill in App name, User support email, Developer contact email
3. Click Save and Continue through Scopes
4. On Test Users step, add your own Google account email address
5. Save and Continue → Back to Dashboard
4
Create OAuth credentials
Go to APIs & Services → Credentials → Create Credentials → OAuth 2.0 Client IDs.
Application type: Desktop app. Name it anything. Click Create.
On the confirmation dialog, click Download JSON.
5
Place the file
Move the downloaded JSON to AI personal files\. Leave the filename as-is.
6
Confirm your config & restart
"google": {
    "enabled": true,
    "client_secret_glob": "AI personal files/client_secret_*.json",
    "token_path": "data/google_token.json",
    "calendar_enabled": true,
    "gmail_enabled": true,
    "gmail_send_enabled": false,
    "drive_enabled": true
}
On first startup with Google enabled, EOS opens a browser window asking you to sign in and grant access. After authorizing, a token is saved to data\google_token.json and you won't be prompted again.

If authorization expires: delete data\google_token.json and restart EOS.
Checking integration status
Open the admin panel → Integrations tab. This shows whether the credential file is found, whether the OAuth token is valid, which services are enabled, and live data previews (upcoming calendar events, inbox, recent Drive files).
Controlling Gmail Send
Gmail sending is off by default. To enable it at runtime without restarting: go to Control & Permissions → Capabilities → Google → Gmail Send. To enable it permanently, set "gmail_send_enabled": true in config.json.
🤖
Need a walkthrough? Copy this prompt and paste it into any AI assistant:
"Walk me through setting up a Google Cloud project for OAuth2 access to Gmail, Calendar, and Drive on a desktop app — I need to enable the APIs, configure the consent screen for personal use, create a Desktop OAuth client ID, and download the credential JSON file. Step by step please."
⚠️
Keep credential files private
The OAuth JSON grants the ability to request access to your Google account. If it leaks, delete and recreate the OAuth client ID in Cloud Console. The token file (data\google_token.json) contains a live refresh token — if it leaks, revoke it at myaccount.google.com/permissions.

Computer Use

EOS can launch and interact with approved desktop applications. This is a layered, opt-in system — you decide which apps are allowed and what actions are permitted within each.

Operating modes
Off
No computer use permitted regardless of the enabled flag.
Command Only
Approved apps only, for explicit user-requested tasks. Recommended starting point.
Supervised Session
Approved apps with bounded continuation while you watch. More autonomous — stay present.
Enabling computer use
1
Enable in the admin panel
Go to Control & Permissions → Capabilities → Computer Use. Toggle Enabled on and set Default Mode to Command Only or Supervised Session.
2
Or set it in config.json
"computer_use": {
    "enabled": true,
    "default_mode": "command_only"
}
Approving applications
The application allowlist lives in data\computer_use\approved_shortcuts\. Each .json file represents one approved app. Three examples are pre-loaded (Notepad, VS Code, Browser).
To add an app
1. Create a new .json file in that folder using the schema in the README.md there
2. Ensure a matching policy entry exists in data\computer_use\app_policy.json
3. In the admin panel, click Computer Use → Reload from disk
To revoke an app
Delete its .json file from the approved_shortcuts folder. Takes effect immediately on next policy check.
HALT and confirmations
⛔
HALT button
The admin panel Computer Use tab has a red HALT button. Pressing it immediately stops all active computer-use activity and sets mode to Off. Use this if EOS does something unexpected.
✅
Pending confirmations
Actions marked soft_confirm or hard_confirm in the app policy require your approval before EOS proceeds. A dialog appears in the Computer Use tab — click Confirm or Deny.

Workspace

EOS has a persistent private workspace at data\workspace\. It survives across sessions and reboots — think of it as EOS's local disk.

📋
context/
Files you place here appear passively in EOS's system prompt every turn. Drop background material, project briefs, or reference docs here.
🔨
projects/
Ongoing work — code, documents, research. EOS creates and modifies files here during active tasks.
📝
notes/
EOS's personal notes and observations across sessions. Persists between conversations.
🗒
scratch/
Temporary working files. Safe to delete manually at any time.
💡
How to share files with EOS
Drop any file into data\workspace\context\. EOS will see it automatically in every turn — no need to mention it. This is the best way to share background material or reference documents.
Workspace permissions
By default, EOS can read and write workspace files. Delete and Execute require the corresponding capabilities to be enabled in the Capabilities panel or config.json (allow_delete and allow_exec).

Worldview System

Give EOS a curated understanding of who you are — your values, reasoning style, priorities — without waiting for it to infer all of that from conversation over time.

You deposit source documents (essays, notes, reflections — anything you've written) and ask EOS to extract a structured profile from them. That profile is injected into every conversation as a compact orientation signal. EOS uses it for interpretive calibration — it shapes how EOS reads your intent, not what it says back to you.
How to use it
1
Place your documents
Put any documents you want EOS to understand you through into data\worldview\sources\. Any file format works — plain text and markdown work best.
2
Ask EOS to process them
Tell EOS: "Update the worldview profile" or "I've added new materials to worldview/sources — process them."
3
EOS integrates the signal
EOS reads the new documents, reads the existing profile (if any), and produces an updated data\worldview\profile.md that integrates the new signal without discarding prior understanding.
How it works
🔇
Silent by default
Dropping a file into sources/ is a silent event. EOS will not acknowledge it until you ask it to process the new material.
🎯
Shapes interpretation
EOS does not quote the profile back at you. It uses it to shape how it reads your intent. Your expressed view in conversation always overrides the profile.
🔄
Manual extraction
There is no automatic extraction. Run it whenever you add new source documents or want to refresh EOS's model of you.

Power User Guide

Canonical Config

EOS uses a single primary config file: config.json at the root of the EOS folder. It represents the full intended system.

📄
One file only
There are no separate profile configs or environment variants. config.json is the single source of truth.
🎛
Runtime overrides
Most capability flags can be changed live in the admin panel without restarting. Changes reset to config.json values on next boot.
🔍
Discovery-first
EOS probes what is actually running rather than assuming all configured services are available. Missing backends degrade gracefully.
💡
The admin Config tab
The admin panel → Config tab shows the full loaded config.json as a read-only view. Use it to inspect what is currently active without opening the file.
Power User Guide

CLI Reference

eos.py does not launch model servers. It only discovers running services, builds the capability map, and starts the WebUI.

python eos.py                      # discover services, print summary, start WebUI
python eos.py --status             # print discovery + capability map only (no WebUI)
python eos.py --port 9000          # override WebUI port (default: 7860)
python eos.py --host 0.0.0.0      # bind WebUI to all interfaces (LAN access)
python eos.py --config config.json # explicit config file path
Start a single backend manually
python -m runtime.server_launcher main --accel gpu --config config.json
python -m runtime.server_launcher tools --accel cpu --config config.json

Config Deep-Dive

The key config blocks in config.json and what each field does.

autonomy_defaults
Controls the starting autonomy state at boot. Each dimension can be toggled live in the admin panel without a restart.
"autonomy_defaults": {
    "perception": true,   // observe environment (screen capture, sensors)
    "cognition":  true,   // background thinking, reflection, idle cognition
    "action":     true,   // tool use and external integrations
    "initiative": true    // proactive, unsolicited action
}
For a passive assistant, set action and initiative to false.
servers
Each server block defines its endpoint, binary paths, model location, and layer counts. The n_gpu_layers field controls whether the GPU or CPU binary is used by the launcher.
"servers": {
    "main": {
        "host": "127.0.0.1",
        "port": 8080,
        "binary_cpu": "binaries/llama-server-cpu.exe",
        "binary_gpu": "binaries/llama-server-cuda.exe",
        "model_path": "models/main/...",
        "n_gpu_layers": 35,
        "cpu_n_gpu_layers": 0,
        "gpu_n_gpu_layers": 35
    }
}
computer_use
"computer_use": {
    "enabled": true,
    "default_mode": "command_only",      // off | command_only | supervised_session
    "approved_shortcuts_path": "data/computer_use/approved_shortcuts",
    "app_policy_path": "data/computer_use/app_policy.json",
    "decision_ring_size": 200
}
Mode can be changed at runtime from the admin Computer Use tab or from Capabilities. The shortcut allowlist and policy file are reloaded live from the admin panel without restart.
workspace_tools
"workspace_tools": {
    "enabled": true,
    "workspace_root": "data/workspace",
    "allow_delete": true,   // can delete files in workspace
    "allow_exec": true      // can execute shell commands
}
allow_delete and allow_exec can be toggled live from the Capabilities panel.
creativity
"creativity": {
    "enabled": true,
    "injection_frequency": "medium",     // off | low | medium | high
    "intensity": "balanced",             // conservative | balanced | exploratory | aggressive
    "invocation_domains": {
        "reasoning_assistance": true,
        "autonomous_idle": true,
        "explanation_generation": true,
        "brainstorming_design": true,
        "stuck_state_recovery": true
    }
}
enabled, injection_frequency, and autonomous_idle can be toggled live from Capabilities. Intensity presets control sampling parameters. temperature, top_p, top_k in the advanced block override the preset if set.
google
"google": {
    "enabled": true,
    "client_secret_glob": "AI personal files/client_secret_*.json",
    "token_path": "data/google_token.json",
    "calendar_enabled": true,
    "gmail_enabled": true,
    "gmail_send_enabled": false,   // off by default — toggle live in Capabilities
    "drive_enabled": true
}
gmail_send_enabled can be toggled live. For individual service flags (calendar_enabled, etc.), edit config.json and restart.
discord

    "respond_only_to_mentions": true,   // set false to respond to all messages

Runtime Discovery & Fallback

At bootstrap, EOS probes each expected service from config.json, classifies them, and builds an effective capability map.

Service states
active
Active
Service is running, responding, and passing health checks.
degraded
Degraded
Service is running but health check indicates reduced capability.
unavailable
Unavailable
Service is not running. Fallback behavior applies.
Example startup output
Main model:       active
Tool helper:      active
Thinking helper:  unavailable (fallback to main)
Creativity:       unavailable (fallback to main)
Vision:           unavailable
STT:              active
TTS:              active

Effective capabilities:
  chat:       available
  tools:      available
  reasoning:  degraded
  creativity: degraded
  vision:     unavailable
  voice:      available
Fallback behavior
Missing backendEffect
Main modelChat unavailable — EOS cannot start
Tool helperTool routing falls back to the main model
Thinking helperReasoning falls back to the main model
Creativity helperCreativity requests fall back to main / degrade cleanly
Vision helperVision features are cleanly disabled
STT modelVoice input unavailable; text chat unaffected
TTS modelVoice output unavailable; text chat unaffected
Discord credentialDiscord integration skipped silently at startup
Google credentialGoogle integration skipped silently at startup

Reference

What is EOS

EOS (Entity Operating System) is a local AI system that runs one or more GGUF language models via llama.cpp. It is Windows-only.

🏠
Fully local
No cloud API required. All models run on your hardware. Nothing leaves your machine — conversation history, memory, and identity all stay local.
🧠
Persistent identity
EOS does not reset between conversations. It maintains a persistent identity, accumulated memory, and an ongoing model of you across all sessions.
🎛
Layered autonomy
Four independently controllable dimensions gate what EOS can do. You decide whether it can perceive, think, act, and take initiative — all adjustable live.
🔌
Modular backends
Main model, tool helper, thinking helper, creativity helper, and vision helper are each separate processes. Missing helpers degrade gracefully.
How it works at a high level
1
Start the backend servers
Use per-role launchers or bundle launchers to start whichever model servers you want.
2
Start EOS
Run start-eos.bat. EOS probes each expected service, performs health checks, and builds a runtime capability map.
3
Use the WebUI
Open http://127.0.0.1:7860/ in your browser. The admin panel at /admin is the operator's control point for all runtime controls.

⚠️
Vision requires a GPU
The vision helper (start-vision-gpu.bat) requires an NVIDIA GPU with sufficient VRAM. There is no CPU-only vision launcher. Vision is an additive capability — start it alongside whichever bundle you normally use.
Model file requirements for vision
models\vision\ must contain both the main .gguf model file and a matching mmproj*.gguf projection file. If either is missing, vision will show as unavailable even with the GPU launcher running.

Capability Degradation

When optional backends or credentials are missing, EOS degrades gracefully rather than failing to start.

MissingEffect
Main modelChat unavailable — EOS will not start without this
Tool helperTool calls fall back to the main model
Thinking helperBackground reasoning falls back to the main model
Creativity helperCreativity requests fall back to the main model
Vision helperVision features are cleanly unavailable
STT modelVoice input unavailable; text chat unaffected
TTS modelVoice output unavailable; text chat unaffected
Discord credentialDiscord integration skipped silently at startup
Google credentialGoogle integration skipped silently at startup
