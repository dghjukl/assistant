Capabilities

Workspace

EOS has a persistent private workspace that survives across sessions and reboots. Think of it as EOS's local disk.

Overview

What the Workspace Is

The workspace lives at data\workspace\. EOS can read, write, and — if permitted — delete files and run shell commands there. This is where EOS does its work: writing code, taking notes, managing projects.

Nothing in the workspace is shared externally unless you explicitly ask EOS to do so. Files accumulate across sessions; EOS remembers what it has written and can return to prior work without you needing to re-explain context.

Layout

Directory Structure

data/workspace/ ├── context/ files here appear in EOS's system prompt every turn ├── projects/ ongoing work — code, documents, research ├── notes/ EOS's personal notes across sessions └── scratch/ temporary working files

context/

Anything placed in context\ is referenced in EOS's system prompt on every turn. Use this for documents you want EOS to always have in view — project briefs, coding standards, background material. EOS does not need to be told about these files; it sees them automatically.

projects/

Long-running work lives here. EOS organises active work into subdirectories — one per project. Code, research notes, drafts, and plans accumulate here across sessions so EOS can pick up where it left off.

notes/

EOS's personal notes — observations, reminders to itself, things it wants to remember between conversations. These are written by EOS, not by you, though you can read and edit them freely.

scratch/

Temporary working files: intermediate outputs, drafts under construction, data being processed. EOS treats this as disposable space. You can clear it at any time without affecting projects or notes.

Giving EOS Files

Sharing Files with EOS

The simplest way to give EOS a file is to drop it into data\workspace\context\. EOS sees files in that folder referenced in its system prompt automatically — you do not need to mention them or paste their contents into the chat.

This is ideal for:

  • Project briefs and requirements
  • Reference documents and specifications
  • Background material EOS should always have
  • Coding standards or style guides

For one-off files that are only relevant to a single conversation, you can paste content directly into the chat instead.

💡
The easiest persistent context
The context/ folder is the simplest way to give EOS persistent background information. Files placed here are seen every turn without you having to mention them.
Access Control

Workspace Permissions

Operation Default How to change
Read files Always allowed Cannot be disabled
Write files Always allowed Cannot be disabled
Delete files Off Enable Allow Delete in Capabilities, or set in config.json
Execute shell commands Off Enable Allow Exec in Capabilities, or set in config.json

To change delete or exec permissions, go to Admin Panel → Control & Permissions → Capabilities → Workspace, or edit config.json:

"workspace_tools": {
  "enabled": true,
  "allow_delete": false,
  "allow_exec": false
}
⚠️
Shell execution
allow_exec enables EOS to run shell commands in the workspace. Only enable this if you have a specific need for it — it grants significantly broader capability than read/write access alone.
Help

Troubleshooting

Symptom Fix
EOS can't see a file you added Make sure the file is in context\, not the workspace root. Only files in context/ are injected into the system prompt automatically.
EOS can't write files Check that "enabled": true is set under "workspace_tools" in config.json.
Delete operation failing Allow Delete must be enabled in Capabilities or "allow_delete": true set in config.json.
Shell exec failing Allow Exec must be enabled in Capabilities or "allow_exec": true set in config.json.
