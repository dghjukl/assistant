Cognition

The Cognition Panel

Located at Admin Panel → Cognition tab. Shows what EOS was actually doing during each turn — the full internal trace, not just the output.

Sub-tab What it shows Available for
Turn Trace Every conversation turn: model used, tools invoked, retrieval results, escalations, and timing for each step. Every turn
Memory Influence Which memory items were retrieved from vector memory and injected into the context for each turn. Every turn
Reflection Events Outputs from background reflection loops — conclusions EOS reached and improvement suggestions it generated. Background events
State Changes Persistent identity state before and after each turn, plus any new memory entries added during the turn. Every turn
💡
Click to expand any turn
Click any turn row to expand a full detail view. This is the best way to understand why EOS responded a certain way — you can see every retrieval, tool call, and model decision that contributed to the output.
Diagnostics

Status Check

Two ways to check the current system state without opening the admin panel.

Method Command
Batch script status-eos.bat
Python CLI python eos.py --status

Both produce the same output: each service listed as active, degraded, or unavailable, followed by the effective capability map derived from what was discovered.

Example Output

Services
  Main model       active        (port 8080)
  Tool helper      active        (port 8082)
  Thinking helper  degraded      (managed on-demand; currently inactive)
  Vision           unavailable
  STT              active
  TTS              active

Effective Capabilities
  chat             available
  tools            available
  reasoning        available     (elastic auxiliary reasoning available on demand)
  vision           unavailable
  voice            available

In the example above, the thinking helper is intentionally dormant. EOS can still activate it later if the executive decides the task justifies escalation and the current system resources allow it. Voice is available because both STT and TTS are active.

Diagnostics

Diagnostic Commands

These commands are for targeted diagnosis when something is not behaving as expected.

Check capability map without starting the WebUI

Use this to quickly probe service availability from a terminal without bringing up the full interface. Safe to run at any time.

python eos.py --status

Start a single backend server explicitly

Use this when you want to bring up one backend manually — for example to test a specific GPU/CPU configuration, or when a launcher script is not available. Replace main with the server name, and adjust --accel and --config as needed.

python -m runtime.server_launcher main --accel gpu --config config.json

Validate Python syntax across the codebase

Use this after editing source files, or when an import error is hard to trace. Compiles every file and reports any syntax errors found, without running any code.

python -m compileall eos.py runtime services webui interfaces core tools verify.py
Diagnostics

Exporting a Diagnostic Bundle

Navigate to Admin Panel → Export tab and click Generate Diagnostic Bundle.

The bundle collects the following into a single JSON file:

  • Current runtime state snapshot
  • Status of all inference servers
  • Recent log entries from the live log stream
  • A snapshot of the loaded config.json

The file is saved to the exports/ folder inside your EOS directory and simultaneously offered as a browser download. Use this when filing a bug report or diagnosing a persistent issue that is difficult to reproduce interactively.

💡
For bug reports
Always attach a diagnostic bundle when reporting an issue. It gives full system context — service states, config, and recent log activity — without requiring manual reproduction steps.
Diagnostics

Memory Panel

Navigate to Admin Panel → Memory tab to inspect the vector memory system.

The Memory tab shows:

  • Vector memory stats — total stored memories and ChromaDB collection size
  • Memory log — a rolling log of recent store and retrieve operations, with timestamps and content summaries

Use the Memory panel to verify that memories are being created after conversations and retrieved correctly at the start of new turns. If the stored count is not growing over time, memory writes may be failing — check the Live Logs tab for related errors.

ⓘ
Storage location
ChromaDB stores embeddings in data\memory_store\. This folder grows over time as more memories are stored. It is safe to leave it in place — ChromaDB manages its own internal compaction. Do not delete this folder unless you intend to wipe all of EOS's long-term memory.
