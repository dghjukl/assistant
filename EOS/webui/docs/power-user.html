Configuration

The Canonical Config

EOS uses a single config file: config.json. It describes the full intended system.

You don't switch between multiple config variants or maintain separate files per environment. The intended workflow is:

  • Launch the backends you want running
  • Let eos.py discover what is available at startup
  • Rely on runtime fallback for missing helpers
  • Use the admin panel for live adjustments during a session
💡
The other config files are reference snapshots
The files configs/profiles/config.base.json, configs/profiles/config.standard.json, and similar variants are reference snapshots — they document known-good configurations for different hardware profiles. config.json is the file EOS actually loads. Edit config.json directly; the snapshot files are read-only references.

CLI Reference

All supported command-line flags for eos.py.

# Discover services, print summary, and start the WebUI (normal startup)
python eos.py

# Print discovery + capability map only — no WebUI started
python eos.py --status

# Override the WebUI port (default: 7860)
python eos.py --port 9000

# Bind to all interfaces for LAN access
python eos.py --host 0.0.0.0

# Explicitly specify the config file path
python eos.py --config config.json
ⓘ
eos.py never starts model servers
eos.py is the coordinator and WebUI host. It discovers running backends but does not start them. Use a launcher script or runtime.server_launcher to bring up model servers before running eos.py.

Launcher Architecture

Launchers are organized into three categories. Mix and match freely to build the stack you want.

Per-Server Launchers

Each launcher starts exactly one backend. Use these when you want precise control over which models run and on which hardware.

  • launchers\start-main-gpu.bat / launchers\start-main-cpu.bat — main language model (port 8080)
  • launchers\start-vision-gpu.bat / start-vision-cpu.bat — vision helper (port 8081)
  • launchers\start-tools-gpu.bat / launchers\start-tools-cpu.bat — tool helper (port 8082)
  • launchers\start-thinking-gpu.bat / launchers\start-thinking-cpu.bat — thinking helper (port 8083)
  • launchers\start-creativity-gpu.bat / launchers\start-creativity-cpu.bat — creativity helper (port 8084)

Bundle Launchers

Each bundle launcher starts a predefined set of backends in one click.

  • launchers\start-minimal.bat — main + tools (minimum viable stack)
  • launchers\start-standard.bat — main + tools + thinking (recommended for most users)
  • launchers\start-full.bat — main + tools + thinking + creativity (full stack, highest resource use)

Bootstrap & Status

  • start-eos.bat — runtime discovery + WebUI (run after backends are up)
  • status-eos.bat — discovery only, no WebUI started

GUI Launcher

  • Launch EOS.bat — graphical launcher with profile selection, starts backends and EOS in the correct order
💡
Mix GPU and CPU freely
You can run the main model on GPU and the tool helper on CPU simultaneously — just use the corresponding per-server launchers. There is no requirement for all backends to use the same hardware. A common pattern is main on GPU for speed, helpers on CPU to conserve VRAM.

Config Deep-Dive

Key blocks in config.json and what each field governs.

servers block

Defines each backend: its endpoint URL, the path to the llama binary, the model file location, and layer counts. The n_gpu_layers field is the key control — set it to 0 for full CPU inference, or to a positive integer to offload that many transformer layers to the GPU. EOS uses this value to select the correct binary (GPU build vs CPU build) at launch time.

"servers": {
  "main": {
    "endpoint": "http://127.0.0.1:8080",
    "binary_gpu": "bin\\llama-server-cuda.exe",
    "binary_cpu": "bin\\llama-server.exe",
    "model": "models\\primary\\model.gguf",
    "n_gpu_layers": 35,
    "context_size": 8192
  },
  "tools": {
    "endpoint": "http://127.0.0.1:8082",
    "binary_gpu": "bin\\llama-server-cuda.exe",
    "binary_cpu": "bin\\llama-server.exe",
    "model": "models\\tools\\model.gguf",
    "n_gpu_layers": 0,
    "context_size": 4096
  }
}

autonomy_defaults block

Sets the starting state of the four autonomy dimensions when EOS boots. All four default to true. Admin panel changes override these at runtime but reset to these values on restart.

"autonomy_defaults": {
  "perception": true,   // observe environment passively
  "cognition": true,    // run background reflection loops
  "action": true,       // use tools and external integrations
  "initiative": true    // act proactively without being asked
}

computer_use block

enabled is the master safety toggle — computer use cannot occur when this is false regardless of mode. default_mode sets how much supervision EOS operates under when computer use is active: command_only requires explicit user requests with per-action confirmation; supervised_session allows a bounded sequence of actions before pausing for review. approved_shortcuts_path points to the folder of approved application shortcuts that EOS is permitted to launch.

"computer_use": {
  "enabled": true,
  "default_mode": "command_only",
  "approved_shortcuts_path": "data\\computer_use\\approved_shortcuts",
  "app_policy_path": "data\\computer_use\\app_policy.json"
}

workspace_tools block

Controls what EOS can do inside the workspace directory. Both flags default to false. allow_delete permits permanent file deletion within the workspace. allow_exec permits EOS to run shell commands — only enable this if you explicitly need scripted automation.

"workspace_tools": {
  "allow_delete": false,
  "allow_exec": false
}

creativity block

injection_frequency controls how often creative reframes are injected: "off", "low", "medium", or "high". intensity controls how strongly the reframes influence the response (float 0.0–1.0). invocation_domains lists the conversation domains where creativity injection is active — remove a domain to suppress creative reframes in that context. autonomous_idle allows the creativity subsystem to run between conversations during idle cognition cycles.

"creativity": {
  "enabled": true,
  "injection_frequency": "medium",
  "intensity": 0.6,
  "autonomous_idle": false,
  "invocation_domains": [
    "problem_solving",
    "creative_writing",
    "brainstorming",
    "open_ended_questions"
  ]
}

google block

Controls which Google Workspace services EOS uses. calendar_enabled, gmail_enabled, and drive_enabled gate each service independently — set any to false to disable that service even when credentials are present.

"google": {
  "calendar_enabled": true,
  "gmail_enabled": true,
  "drive_enabled": true
}

Runtime Discovery

At startup, eos.py probes each expected service and builds a live capability map before opening the WebUI.

The discovery process:

  1. Read the servers block from config.json to get expected endpoints
  2. Probe each endpoint with a lightweight health check
  3. Classify each service as active, degraded, or unavailable
  4. Build the effective capability map using the fallback rules below
  5. Print the summary to the console and make it available in the Admin Panel Overview tab

Fallback rules applied during discovery:

  • Tool helper missing → tool calls fall back to main model (degraded)
  • Thinking helper missing → reasoning falls back to main model (degraded)
  • Creativity helper missing → creativity falls back to main model or degrades cleanly
  • Vision helper missing → vision features cleanly disabled (unavailable)
  • STT/TTS missing → voice degrades without blocking chat

Data Directory Layout

Complete layout of the data/ directory, with each file and folder explained.

data/
├── entity_state.db                 # SQLite: persistent identity state and long-term memory
├── google_token.json               # Google OAuth token (auto-created on first auth)
├── memory_store/                   # ChromaDB vector memory — grows over time, do not delete
│   └── ...                         # ChromaDB internal files managed automatically
├── workspace/                      # EOS's sandboxed file environment
│   ├── context/                    # Files here are auto-injected into EOS's context
│   ├── output/                     # Files EOS writes during sessions
│   └── scratch/                    # Temporary working files
├── worldview/                      # Worldview personalization subsystem
│   ├── sources/                    # Drop source documents here for worldview extraction
│   ├── profile.md                  # Extracted worldview profile applied to EOS's context
│   └── extraction_log.json         # Log of worldview extraction runs
├── computer_use/                   # Computer use subsystem data
│   ├── approved_shortcuts/         # Shortcuts EOS is permitted to launch
│   └── app_policy.json             # Application permission policy
└── backups/                        # Automatic state snapshots (created on major state changes)
💡
Backing up EOS state
To fully back up EOS, copy the entire data/ directory and config.json. The data/ directory contains everything persistent: identity, memory, OAuth tokens, worldview, and workspace files. Model files in models/ do not need to be backed up — they can always be re-downloaded with setup\Setup-Full.ps1.
